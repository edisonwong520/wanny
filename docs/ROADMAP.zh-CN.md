# Wanny Roadmap

[English Version](./ROADMAP.md)

这份 roadmap 用来整理 Wanny 接下来的主要演进方向。内容会尽量保持务实，重点放在产品价值、工程落地和当前最值得补齐的能力缺口上。

## 近期方向

### 1. 接入更多设备与平台

- 在现有接入基础上继续扩展，持续扩大可连接的真实物理设备范围。
- 优先补齐用户需求强的平台与品牌，比如海信以及更多主流家电厂商。
- 优化 provider 标准化能力，让新平台更容易进入统一的 `provider -> devices -> control -> memory` 链路，而不是每接一个都写大量特例。
- 持续收敛设备命名、能力描述、遥测状态和控制语义，避免把厂商底层字段直接暴露给用户。

### 2. 完成更多测试与稳定性建设

- 继续补齐微信链路、设备查询、设备控制、审批确认、后台持久化等端到端测试。
- 强化 provider 适配层的回归测试，尤其是 token 刷新、状态解析、控制执行和能力映射。
- 完善前端构建与交互相关的验证，尽早发现部署类回归问题。
- 为“能识别”“能控制”“能恢复”建立更明确的稳定性门槛，再把新接入视作真正可用。

### 3. 人文关怀能力

- 把天气、季节和环境上下文转成真正有用的建议，而不只是被动展示状态。
- 支持像“天气转冷了，是否要调整某些家电设置”这类面向生活场景的建议联动。
- 这些建议需要可解释、轻量、克制，不能变成高频打扰或过度自动化。
- 持续把 Wanny 从“能执行命令”往“能照顾日常生活”推进。

### 4. 主动巡检

- 自动发现维护和风险信号，例如滤芯该更换、耗材将用尽、设备遥测异常等。
- 先从价值明确的场景切入，比如饮水机、净化设备、空气系统等维护周期清晰的设备。
- 让主动提醒具备可追溯、可确认、可忽略的能力，而不是单向推送。

## 中期方向

### 5. 用小模型做语义理解

- 逐步减少对硬编码关键字和启发式规则的依赖。
- 用更轻量、更便宜的小模型来完成语义路由、查询分类、实体识别和设备意图理解。
- 保留必要的启发式兜底以提升安全性，但让主判断逐步迁移到模型语义层。
- 建立一套基于真实用户表达的评估集，避免只凭感觉判断识别效果。

### 6. 线程池 / 并发执行能力

- 优化非阻塞任务的执行吞吐，例如记忆写入、向量持久化、上下文补全、provider 同步等。
- 把更多非关键副作用移出主响应链路，同时保留一致性要求高的操作为同步执行。
- 引入更安全的 worker / 线程池式调度与并发边界，避免后台任务扩散成新的隐患。
- 结合 tracing、telemetry 和耗时日志，把延迟和瓶颈可视化。

## 更长期方向

### 7. 登录、交易与账号型操作

- 逐步支持更强的登录态工作流，以及和账号、服务、交易型行为相关的能力。
- 明确区分低风险的信息读取和高风险的账号操作、支付、预订、不可逆动作。
- 这一层必须按高安全级别设计，不能只靠普通聊天执行，需要更严格的审批、审计和身份确认。

### 8. Console 持续完善

- 持续把 Console 从“管理面板”推进成真正的运营中枢。
- 完善设备总览、平台授权、任务审阅、记忆查看和动作解释能力。
- 提升信息密度、跨平台可见性，以及失败后的恢复体验。
- 让用户能更清楚地知道 Wanny 记住了什么、准备做什么、为什么给出某个建议。

## 横向原则

- 安全优先于自治。
- 真实设备与真实世界副作用必须可解释、可审计。
- 新平台接入应尽量复用统一抽象，而不是沉淀越来越多的一次性特例。
- 用户信任来自稳定、克制和明确的恢复路径。
